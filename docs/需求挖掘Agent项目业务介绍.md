# 需求挖掘 Agent 项目业务介绍

> 项目名称：需求挖掘 Agent  
> 所属业务：好游快爆工具与体验服业务  
> 负责团队：拓展组  
> 文档定位：面向业务负责人、产品、运营及协作团队的项目介绍

---

## 一、项目概述

需求挖掘 Agent 是一套面向好游快爆业务的智能需求发现与管理系统。

系统通过持续监控站内外游戏用户讨论，从大量帖子、视频和社区内容中识别用户反复提出的问题、未被满足的需求以及正在升温的业务机会，并将分散内容整理为可以直接评估的需求卡片。

项目当前重点服务两类业务：

1. **游戏工具需求**：发现适合通过工具、查询、计算、地图、数据或攻略辅助能力解决的用户问题；
2. **体验服需求**：发现体验服更新、爆料、资格招募、开放时间及当前测试节点。

项目希望解决的核心问题不是“找到热门内容”，而是从热门和非热门内容中识别：

- 用户究竟遇到了什么问题；
- 这个问题是否反复出现；
- 当前解决方式是否足够麻烦；
- 是否适合通过快爆业务能力解决；
- 需求是否值得进入评估或立项；
- 需求出现后是否正在持续升温。

## 二、项目背景

游戏用户的真实需求通常不会以标准需求文档的形式出现，而是分散在不同平台的提问、吐槽、攻略讨论、搜索行为和自制内容中。

例如：

- “战备值怎么配才够？”
- “有没有可以直接算材料的工具？”
- “新地图资源点太散，求一个地图标记。”
- “抽卡记录在哪里分析？”
- “体验服资格什么时候开放？”
- “新版本到底更新了什么？”

过去主要依靠业务人员人工搜索、阅读和判断，存在以下问题：

### 2.1 需求分散

同一个问题可能同时出现在好游快爆、TapTap、抖音、小黑盒等多个平台，单个平台的数据无法完整反映用户需求。

### 2.2 人工判断成本高

业务人员需要阅读大量无关内容，才能找到少量有价值的需求线索，且结果容易受到个人经验和关注范围影响。

### 2.3 热度不等于需求

热门内容可能只是资讯、娱乐或营销内容；一些热度不高但反复出现的问题，反而更适合建设长期工具。

### 2.4 缺乏统一证据

需求被提出后，往往缺少用户原始内容、来源链接、讨论频率和外部方案等证据，导致后续评估依赖主观判断。

### 2.5 缺少持续跟踪

需求发现、评估、开发和上线分散在不同流程中，难以持续观察需求变化，也不容易沉淀历史经验。

### 2.6 体验服信息时效短

体验服资格、版本更新和爆料具有明显的时间窗口，人工发现较慢时，业务机会可能已经过期。

## 三、项目目标

### 3.1 核心目标

1. 持续发现具有业务价值的用户需求；
2. 降低业务人员搜索和整理需求的人工成本；
3. 为每条重点需求提供可追溯的用户证据；
4. 给出需求价值、可行性和优先级参考；
5. 识别快爆站内已有能力，减少重复建设；
6. 对重点需求和体验服节点进行持续监控；
7. 沉淀需求从发现到上线的完整记录。

### 3.2 项目不替代的工作

需求挖掘 Agent 提供需求发现和决策依据，但不替代：

- 业务负责人进行最终立项决策；
- 产品经理完成具体产品方案；
- 研发团队进行技术可行性评估；
- 运营人员制定完整运营计划；
- 内容团队判断具体内容表达方式。

系统输出的评分和建议属于辅助信息，不代表需求必须立项。

## 四、目标使用者

| 使用者 | 主要用途 |
|---|---|
| 业务负责人 | 查看高价值需求，确定业务优先级 |
| 产品经理 | 获取用户问题、使用场景和产品机会 |
| 运营人员 | 发现热点节点、资格信息和内容机会 |
| 工具业务人员 | 判断新工具立项或已有工具优化方向 |
| 体验服业务人员 | 跟踪更新、爆料、招募和测试节点 |
| 内容团队 | 发现用户高频问题和攻略选题 |
| 管理人员 | 查看需求趋势、业务沉淀和处理进度 |

## 五、业务价值

### 5.1 从“人工找需求”变为“系统推送需求”

业务人员不需要每天在多个平台重复搜索。系统按照监控对象和关键词自动采集内容，并将重点需求集中展示。

### 5.2 从“单条内容”变为“需求主题”

系统不会简单地把每一篇帖子当成一条需求，而是将相近问题整理为需求主题，帮助业务判断需求是否具有普遍性。

### 5.3 从“主观感觉”变为“证据辅助判断”

每条需求可同时展示用户原始内容、来源平台、讨论热度、需求信号和分析理由，使业务判断更加透明。

### 5.4 从“发现新需求”扩展到“优化已有业务”

当需求与快爆站内已经上线的工具匹配时，系统会提示优先评估：

- 用户是否不知道已有工具；
- 已有工具是否需要更新；
- 当前能力是否只解决了部分问题；
- 是否需要改善入口和触达。

### 5.5 从“一次性调研”变为“持续监控”

系统保存每日需求和历史状态，可以观察需求是短期热点、持续增长，还是已经逐渐消退。

## 六、项目业务流程

需求挖掘 Agent 当前按照以下流程运行：

```text
维护监控游戏与搜索词
        ↓
采集站内外用户内容
        ↓
清洗、去重并关联游戏
        ↓
计算需求信号
        ↓
提炼用户问题和需求主题
        ↓
匹配快爆已有工具
        ↓
生成需求卡片和每日总结
        ↓
业务评估并更新处理状态
        ↓
进入历史沉淀和持续监控
```

### 6.1 监控配置

业务人员首先维护需要关注的游戏、游戏状态和优先权重，同时为不同平台配置搜索关键词。

系统仅对“热门”“在运营”和“测试中”的游戏持续挖掘需求；已停运游戏不再产生新需求，但保留历史记录。

### 6.2 数据采集

系统根据已配置的平台、关键词和抓取数量采集用户内容，同时接收Tap+快爆论坛监控后台的增量数据。

### 6.3 数据清洗

采集到的内容会经过：

- 游戏关联；
- 无关内容过滤；
- 重复内容去除；
- 来源和时间标准化；
- 采集失败及数量不足原因记录。

### 6.4 需求信号计算

系统从多个维度判断某款游戏是否正在产生值得关注的需求。

### 6.5 需求主题提炼

系统结合规则和智能分析，将用户内容整理为明确的需求主题，并生成标题、描述、判断理由和建议方向。

同一款游戏可以同时产出多个不同主题的需求。

### 6.6 业务评估

业务人员查看需求证据和分析结果后，可以将需求更新为待评估、已采纳、开发中、已上线或已驳回。

### 6.7 持续跟踪

需求会进入历史排行榜和需求管理模块，方便后续查询、对比和复盘。

## 七、当前需求范围

### 7.1 游戏工具需求

当前可以重点识别以下方向：

| 工具方向 | 典型用户问题 |
|---|---|
| 配装/战备工具 | 装备怎么搭配、战备值如何达到要求 |
| 交互地图 | 资源点、宝箱、撤离点和收集物在哪里 |
| 抽卡/概率分析 | 抽卡记录、保底和概率如何分析 |
| 资格/福利聚合 | 资格、兑换码、礼包码和活动入口在哪里 |
| 机制计算器 | 材料、伤害、收益和游戏机制如何计算 |
| 排行榜/对战数据 | 战绩、排名和对局数据如何查询 |
| 剧情/收集进度 | 收集进度、遗漏内容和完成情况如何管理 |
| 攻略辅助 | 信息分散且需要结构化整理的攻略问题 |
| 模拟器 | 用户需要提前模拟选择或结果 |
| 数据库 | 物品、角色、价格和机制数据集中查询 |

系统也可以发现尚未归入固定分类的新方向。

### 7.2 体验服需求

体验服需求不作为普通工具需求展示，而是单独整理以下内容：

1. **更新内容**：版本新增、玩法调整、平衡改动和测试公告；
2. **爆料内容**：新角色、新地图、新武器和新玩法等信息；
3. **资格招募**：报名入口、申请条件、抢码和资格发放；
4. **招募时间**：明确日期、时间或开放周期；
5. **当前节点**：观察中、内容爆料、版本更新、预告中、报名中或已结束。

## 八、需求判断方法

### 8.1 当前需求信号

系统当前使用七类信号辅助判断：

| 信号 | 说明 |
|---|---|
| 重复提问密度 | 相同或相近问题是否被多人反复提出 |
| 信息分散度 | 用户是否需要跨页面、跨攻略或跨平台寻找答案 |
| 民间工具萌芽 | 是否出现用户自制表格、文档、网页或工具 |
| 资格稀缺信号 | 是否涉及体验服、限量资格、抢码等稀缺资源 |
| 机制复杂度 | 是否存在大量参数、规则、组合或计算过程 |
| 内容热度 | 相关内容的浏览、点赞、评论和传播情况 |
| 外部平台工具上线 | 外部是否已经出现同类工具或解决方案 |

### 8.2 需求等级

系统根据综合潜力分给出参考等级：

| 等级 | 分数 | 建议 |
|---|---:|---|
| S级 | 85—100 | 优先组织专项评估 |
| A级 | 70—84 | 进入近期需求池 |
| B级 | 50—69 | 持续观察并补充证据 |
| C级 | 0—49 | 暂不推进或调整监控方向 |

### 8.3 工具化可行度

除需求潜力外，系统还会给出工具化可行度，用于区分：

- 用户问题是否可以通过明确功能解决；
- 是否存在可计算、可查询或可结构化的数据；
- 需求是否更适合做工具、内容或运营活动；
- 当前是否具备实现基础。

## 九、当前功能模块

### 9.1 今日需求

展示当天需求挖掘结果，包括：

- 今日需求总量；
- 活跃趋势游戏；
- 高价值需求数量；
- 需求等级和热门工具方向；
- 当日重点建议；
- Tap+快爆新增内容提醒；
- 数据采集和分析进度；
- 手动分析和强制重新抓取。

### 9.2 历史排行榜

将历史需求按照潜力分沉淀，支持查看：

- 历史需求数量；
- S级和A级高价值需求；
- 不同时间范围内的重点需求；
- 需求评分和处理状态。

### 9.3 需求管理

用于统一管理已经发现的需求，支持按照以下条件筛选：

- 游戏；
- 工具类型；
- 工具需求或体验服需求；
- 需求等级；
- 处理状态；
- 日期。

业务人员可以直接修改需求状态并进行后续跟踪。

### 9.4 需求详情

单条需求详情包含：

- 游戏和需求类别；
- 需求标题、描述和潜力分；
- 七维需求信号；
- 用户高频问题；
- 当前信息缺口；
- 分析理由；
- 工具化可行度；
- 代表性证据帖及来源链接；
- 快爆已上线工具匹配；
- 相似历史需求。

体验服需求详情会重点展示更新、爆料、资格和当前节点。

### 9.5 监控数据

用于查看已经采集并存储的平台内容，支持按照平台、时间和关键词进行筛选。

该模块帮助业务人员核对原始数据，判断需求分析结果是否具有足够依据。

### 9.6 搜索词配置

用于管理不同平台的全局搜索关键词、抓取数量和启用状态。

搜索词自动应用于游戏管理中的全部活跃游戏。部分平台可配置代理，以应对访问限制。

### 9.7 游戏管理

用于维护监控游戏，包括：

- 游戏名称；
- 游戏品类；
- 厂商；
- 运营状态；
- 好游快爆内部ID；
- 搜索与需求挖掘权重；
- 业务备注。

重点游戏可以设置更高权重，使其需求获得更高关注。

## 十、标准需求卡片

每条需求卡片主要回答以下问题：

| 问题 | 对应内容 |
|---|---|
| 需求是什么 | 需求标题和描述 |
| 谁遇到了问题 | 游戏及目标用户场景 |
| 为什么值得关注 | 潜力分、等级和需求信号 |
| 是否适合做工具 | 工具化可行度和建议类型 |
| 判断依据是什么 | 用户高频问题、分析理由和证据帖 |
| 快爆是否已有能力 | 已上线工具匹配结果 |
| 以前是否出现过 | 相似历史需求 |
| 当前如何处理 | 需求状态和业务备注 |

需求卡片的目标是让业务人员不需要重新阅读全部原始内容，也能快速完成初步判断；需要进一步核实时，可以通过证据链接返回原始内容。

## 十一、典型业务案例

### 11.1 配装与战备需求

当多名玩家反复询问战备值、武器配件和配装方案，并出现自制配装表时，系统可以将分散讨论整理为“战备/配装工具”需求。

业务可进一步判断：

- 是否建设计算器；
- 是否提供推荐方案；
- 是否整合已有攻略；
- 是否已有站内工具可以更新。

### 11.2 地图和收集需求

当玩家集中讨论新地图资源点、撤离路线、宝箱或收集物位置时，系统可以识别交互地图需求，并保留相关内容作为证据。

### 11.3 兑换码和福利需求

当用户集中搜索最新兑换码、密令、礼包码及有效期时，系统可以将其识别为资格/福利聚合需求。

### 11.4 体验服资格需求

当多个平台出现体验服报名、抢码和资格开放消息时，系统会整理：

- 是否已经开启；
- 报名或开放时间；
- 申请入口相关信息；
- 当前体验服节点；
- 对应的用户证据。

### 11.5 已有工具优化

当新需求与快爆已上线工具匹配时，系统不会简单建议重复建设，而是提示业务优先检查已有工具的覆盖范围、更新情况和用户触达。

## 十二、运行与交付方式

### 12.1 自动运行

系统每日定时执行完整需求挖掘流程，包括数据接入、信号计算、需求提炼和日报生成。

Tap+快爆论坛后台数据以更短周期进行增量同步，新内容会在页面中提示。

### 12.2 手动运行

业务人员可以在今日需求页面手动触发分析。

当日已经完成分析后，系统会提示“今日已挖掘完成”，避免无意重复执行。如确实需要重新采集，可使用强制重新抓取。

### 12.3 标准交付结果

- 每日需求总览；
- 高价值需求清单；
- 单条需求证据和分析；
- 体验服节点信息；
- 历史需求排行榜；
- 需求状态及处理记录；
- 数据采集失败和不足原因。

## 十三、当前数据来源

项目当前已具备或预留以下数据接入能力：

- TapTap；
- 抖音；
- 小黑盒；
- Tap+快爆论坛监控后台；
- 本地模拟数据；
- 后续可接入的其他公开数据或业务自有数据。

实际可用范围受平台登录、访问限制、接口稳定性和业务授权影响。

当数据源不可用、平台返回数量不足或内容被过滤时，系统会记录相应原因，避免将“没有采集到数据”误判为“没有用户需求”。

---

## 十四、当前项目边界

### 14.1 当前聚焦范围

- 监控对象以游戏为主；
- 核心业务为游戏工具和体验服；
- 需求判断依赖公开内容和已接入业务数据；
- 分析结果主要用于内部需求评估；
- 当前由业务人员对需求进行最终确认。

### 14.2 当前限制

1. 不同平台的数据完整性和稳定性存在差异；
2. 部分平台需要登录、代理或人工处理验证；
3. 智能分析可能出现误判，需要业务结合证据审核；
4. 新游戏、别名和体验服名称需要持续维护；
5. 潜力分主要用于排序，不等同于实际收益预测；
6. 需求上线后的业务效果仍需要业务方回流；
7. 当前项目主要服务内部使用，尚未形成面向外部用户的产品。

## 十五、业务使用建议

### 15.1 先明确业务问题

新增监控对象或关键词前，应先说明希望验证的问题，不建议仅使用过于宽泛的词进行全量搜索。

### 15.2 同时关注高分和重复出现

高分需求适合优先查看，但长期反复出现的中等评分需求，也可能具有稳定业务价值。

### 15.3 结合已有能力判断

发现需求后，应先检查快爆是否已经有相同或相近能力，区分新建、优化、更新和加强触达。

### 15.4 及时更新处理状态

业务人员应持续反馈需求是否有效、是否采纳以及驳回原因。反馈越完整，后续需求判断和规则优化越准确。

### 15.5 在重要节点提高监控强度

游戏上线、版本更新、测试开启和大型活动前后，用户需求变化更快，建议增加重点关键词和监控频率。

---

## 十六、后续可扩展方向

在工具和体验服业务验证稳定后，需求挖掘能力可以逐步支持其他业务组，例如：

- 攻略选题挖掘；
- 社区功能和治理需求；
- 福利及活动需求；
- 游戏预约和新游关注；
- 用户增长和召回机会；
- 内容消费和搜索需求；
- 商业化反馈；
- 游戏厂商舆情和用户问题。

扩展时需要由对应业务组提供业务背景、目标用户、关键词、有效需求标准和结果反馈机制，避免直接套用工具业务的判断规则。

---

## 十七、项目总结

需求挖掘 Agent 将分散的用户讨论转化为结构化、可追溯、可评估的业务需求。

它的核心价值可以概括为：

> 更早发现需求，更快看清证据，更统一地完成评估，并持续跟踪需求从出现到落地的全过程。

当前项目已经形成从监控配置、数据采集、需求识别、证据整理、价值排序到状态管理的基本闭环，可作为工具与体验服业务的日常需求发现平台，也为后续支持其他业务组提供基础能力。
