# 游戏仅手动新增设计

## 目标

游戏管理在系统正常运行期间仅允许用户通过游戏管理页面或 `POST /api/games` 手动新增游戏，采集、同步和分析流程不得自动创建游戏。

首次安装且游戏表为空时，继续初始化项目内置的默认游戏，避免空系统无法直接使用。

## 行为边界

### 保留

- 空数据库首次启动时，通过现有种子初始化逻辑创建默认游戏。
- 游戏管理页面手动新增、编辑和删除游戏。
- 外部内容按精确名称、标准化名称或正文提及匹配已有游戏。
- 未匹配内容继续写入外部原始记录，便于后续补建游戏和追溯。

### 禁止

- TapTap、快爆论坛及其他外部同步根据内容中的游戏名自动创建游戏。
- 采集、雷达扫描、每日分析等后台流程绕过游戏管理接口创建游戏。
- 未匹配内容直接进入正式内容池或触发需求分析。

## 数据流

1. 外部同步先保存原始记录。
2. 使用现有匹配规则查找游戏管理中已有的游戏。
3. 匹配成功时写入 `platform_contents`，并进入后续扫描和分析。
4. 匹配失败时增加 `unmatched_games`，不创建游戏、不写入正式内容池。
5. 用户手动新增对应游戏后，后续同步可正常匹配并入库。

## 接口与页面

- `POST /api/games` 保持为唯一的运行期新增游戏入口。
- 游戏管理页面现有“新增游戏”功能保持不变。
- 外部同步状态继续展示“未匹配游戏”数量。
- 移除 `created_games` 运行期统计字段，避免继续表达已废弃行为。

## 实现范围

- 删除外部同步服务中的自动建游戏分支和辅助方法。
- 删除不再需要的自动建游戏相关导入、注释与统计字段。
- 保留 `DataAdapter.seed_games()`，但仅由首次空库初始化调用。
- 更新前端同步状态类型，移除 `created_games`。

## 测试与验收

- 未配置游戏的外部内容不会创建游戏，`inserted=0`、`unmatched_games=1`。
- 可从标题推断出游戏名的内容也不会自动创建游戏。
- 标准化名称仍可匹配已有游戏并正常入库。
- 原始外部记录在未匹配时仍然保留。
- 空库首次初始化仍会创建默认游戏。
- 手动新增游戏接口继续可用。
- 后端全量测试和前端构建通过。
