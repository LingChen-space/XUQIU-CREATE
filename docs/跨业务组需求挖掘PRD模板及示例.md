# 跨业务组需求挖掘 PRD 模板及示例

> 文档用途：供好游快爆各业务组申请需求挖掘支持时填写，作为需求挖掘、价值评估、结果交付和持续监控的共同依据。  
> 适用业务：工具、体验服、攻略、社区、福利、预约、内容、发行、增长、商业化及其他业务。  
> 负责团队：工具与体验服业务组（需求挖掘支持方）。

---

## 一、文档说明

### 1.1 为什么需要提交本 PRD

需求挖掘不是简单搜索关键词，而是围绕明确的业务问题，持续收集用户讨论和市场信号，识别真实需求，并判断其是否值得进入产品、内容或运营流程。

申请业务组需要提供必要的业务知识和判断标准；需求挖掘支持方负责将这些信息转化为监控方案，并完成：

1. 需求线索采集与清洗；
2. 用户痛点、场景和需求主题提炼；
3. 证据链整理、价值评分和优先级建议；
4. 周期性监控、变化提醒和结果复盘。

### 1.2 使用方式

1. 申请业务组填写第二至十三章；
2. 提交前使用第十四章检查清单自查；
3. 双方完成启动评审，确认范围、口径和交付周期；
4. 需求挖掘支持方配置监控并开始执行；
5. 申请业务组评估结果并反馈采纳情况；
6. 双方根据反馈调整挖掘规则，进入持续监控。

### 1.3 填写原则

- 描述需要解决的业务问题，不要只写“帮忙看看用户有什么需求”。
- 区分用户原话、业务判断和待验证假设。
- 明确哪些内容属于有效需求，哪些属于噪声。
- 提供可验证的成功指标，避免使用“效果好”“比较准确”等模糊表述。
- 不确定的内容可以标注为“待验证”，但必须说明希望通过挖掘验证什么。

## 八、需求识别与判断标准

### 8.1 什么是有效需求

有效需求至少应包含下列一项：

- 用户反复提出明确问题或目标；
- 用户当前解决过程成本高、步骤多或信息分散；
- 用户通过表格、文档、脚本或其他方式自发解决；
- 外部产品已经提供同类能力并获得用户使用；
- 需求由版本、活动、政策或市场变化触发，具有明确时效；
- 业务方提供的其他有效信号：【填写】。

### 8.2 什么不是有效需求

- 仅出现业务关键词，但没有用户问题或使用场景；
- 单纯搬运资讯、营销内容或广告；
- 只有情绪表达，无法还原具体问题；
- 与监控对象无关的同名内容；
- 已有能力能够完整解决，且没有优化诉求；
- 业务方指定的其他排除情况：【填写】。

### 8.3 业务特有判断规则

| 规则 | 说明 | 示例 |
|---|---|---|
| 【规则一】 | 【如何判断】 | 【正反例】 |
| 【规则二】 |  |  |

### 8.4 最低证据要求

重点需求原则上至少满足以下一项：

1. 多个独立用户或内容源重复出现；
2. 单条内容热度显著，且问题和场景明确；
3. 已存在用户自制方案或外部成熟方案；
4. 申请业务组确认属于战略性机会；
5. 突发时效需求，经业务负责人确认可快速行动。

### 9.1 需求分类

| 一级分类 | 二级分类 | 说明 |
|---|---|---|
| 【填写】 | 【填写】 | 【填写】 |

### 9.2 通用需求信号

系统可结合以下信号进行分析：

1. **重复提问密度**：相同或相近问题出现的频率；
2. **信息分散度**：用户是否需要跨页面、跨平台寻找答案；
3. **民间方案萌芽**：是否出现用户自制工具、文档、表格或教程；
4. **稀缺与时效性**：是否涉及限量名额、短期活动、版本节点等；
5. **机制或决策复杂度**：用户是否面临大量参数和选择；
6. **内容热度**：相关内容的浏览、互动和传播表现；
7. **外部方案验证**：其他平台或竞品是否已经提供同类能力。

不同业务可调整信号权重，但需要在启动评审时明确。

### 9.3 建议评分模型

| 维度 | 权重建议 | 评分说明 |
|---|---:|---|
| 用户需求强度 | 25% | 提问频次、表达明确度、痛点强度 |
| 用户覆盖范围 | 15% | 涉及用户规模及人群广度 |
| 业务价值 | 20% | 对活跃、留存、转化或内容消费的价值 |
| 解决方案可行性 | 15% | 数据、技术、内容或运营可实现程度 |
| 时效性 | 10% | 是否需要在特定节点快速响应 |
| 竞争与差异化 | 10% | 外部验证及快爆可形成的差异 |
| 证据可信度 | 5% | 来源数量、质量及一致性 |

最终得分为参考，不代替业务决策。

### 9.4 优先级建议

| 等级 | 参考分数 | 处理建议 |
|---|---:|---|
| S级 | 85—100 | 立即组织专项评估，优先进入立项或快速响应 |
| A级 | 70—84 | 进入近期需求池，补充可行性和收益评估 |
| B级 | 50—69 | 持续观察，等待更多证据或合适业务节点 |
| C级 | 0—49 | 暂不推进，保留记录或调整监控规则 |

### 9.5 已有能力匹配

若需求与站内已有产品、栏目或内容匹配，应标记为：

- 已完整满足：不重复立项，关注触达问题；
- 部分满足：评估已有能力优化；
- 已有但过期：评估更新或重构；
- 尚未满足：进入新需求评估。

## 十一、协作流程与责任分工

### 11.1 标准流程

1. **PRD提交**：申请业务组提交完整文档和已知样本；
2. **启动评审**：双方确认范围、关键词、判断标准和交付周期；
3. **方案配置**：需求挖掘支持方配置对象、来源、关键词和规则；
4. **试运行**：通过一轮数据验证召回范围和噪声情况；
5. **首轮交付**：输出需求清单、证据链和优先级建议；
6. **业务评估**：申请业务组确认有效性及处理状态；
7. **规则优化**：根据误判、漏判和采纳结果调整规则；
8. **持续监控**：按约定周期提供趋势、提醒和复盘。

## 十五、工具与体验服业务需求挖掘 PRD 示例

> 本示例根据当前需求挖掘项目的实际能力编写，用于展示填写颗粒度。其他业务组应替换为自身业务目标、规则和指标，不应直接照搬评分权重。

### 15.1 申请摘要

| 项目 | 示例内容 |
|---|---|
| 申请业务组 | 工具与体验服业务组 |
| 需求名称 | 游戏工具与体验服用户需求持续挖掘 |
| 业务负责人 | 工具与体验服业务负责人 |
| 日常对接人 | 需求挖掘项目产品经理 |
| 计划启动时间 | 2026年6月 |
| 期望首轮交付时间 | 完成配置后的首个完整监控周期 |
| 挖掘类型 | 长期监控 |
| 期望交付频率 | 每日扫描、每日结果看板、重点节点即时提醒 |
| 需求紧急程度 | 中；体验服资格和版本爆料等需求具有较强时效性 |
| 关联项目 | 需求挖掘 Agent |

**一句话需求：**

希望通过监控站内外游戏用户讨论，持续发现具有产品化价值的游戏工具需求，以及体验服更新、爆料和资格招募信息，用于支持工具立项、已有工具优化和体验服内容运营。

**预期决策：**

- 判断某项用户问题是否值得建设工具；
- 判断已有站内工具是否需要更新或提升触达；
- 识别体验服当前所处节点及用户关注内容；
- 确定高价值需求的优先级和跟进状态；
- 提前发现外部平台和民间工具的竞争信号。

### 15.2 业务背景与问题

好游快爆为游戏用户提供工具、福利和体验服等服务。游戏数量多、版本变化快，用户需求分散在快爆、TapTap、抖音、小黑盒等不同平台。

当前主要问题：

1. 用户需求通常以提问、吐槽、攻略讨论或自制表格等形式出现，人工发现依赖经验；
2. 相同需求分散在多个平台，难以判断是个例还是普遍问题；
3. 热门内容不一定适合做工具，需要综合判断问题强度、机制复杂度和可行性；
4. 快爆可能已上线相似工具，但需求发现环节缺少自动匹配，容易重复立项；
5. 体验服信息时效短，需要区分更新内容、爆料内容、资格招募和当前节点；
6. 需求发现后缺少统一的证据、评分和状态沉淀方式。

**待验证假设：**

1. 重复提问、信息分散和复杂决策可以有效预测工具需求；
2. 用户自制工具或外部同类工具上线，说明需求已得到初步验证；
3. 工具需求和体验服内容需求需要分别分类和展示；
4. 将需求证据、评分和已有能力匹配放在同一需求卡片中，可以提高业务评估效率。

### 15.3 目标与成功指标

**核心目标：**

1. 每日自动发现与活跃游戏相关的工具和体验服需求；
2. 将原始内容整理为可评估、可追溯的需求卡片；
3. 识别需求潜力、工具化可行性和已有能力匹配情况；
4. 持续记录需求状态，形成历史需求排行榜和趋势；
5. 对体验服资格开放、版本更新和重要爆料进行及时提醒。

**非目标：**

1. 不替代业务负责人进行最终立项决策；
2. 不将所有高热度游戏内容都识别为工具需求；
3. 不自动承诺外部数据源始终完整可用；
4. 不在本系统内直接完成工具研发和内容发布。

**成功指标示例：**

| 指标 | 示例标准 |
|---|---|
| 监控对象覆盖 | 覆盖游戏管理中全部活跃、热门和测试中游戏 |
| 数据源覆盖 | 覆盖已配置且可用的平台及Tap+快爆后台数据 |
| 证据完整性 | 重点需求具有可追溯证据帖或明确说明证据不足 |
| 分类准确性 | 工具需求与体验服需求可分别展示和筛选 |
| 业务可行动性 | 每条重点需求可进入采纳、开发、上线或驳回状态 |
| 时效性 | 每日自动运行；重点资格或节点信息支持及时提醒 |
| 结果沉淀 | 需求进入历史排行榜，并保留评分、证据和处理状态 |

### 15.4 目标用户与场景

| 用户 | 典型诉求 | 典型行为 |
|---|---|---|
| 普通游戏用户 | 快速找到答案，降低信息搜索成本 | 反复搜索、提问、收藏攻略 |
| 进阶玩家 | 计算配装、概率、路线和资源收益 | 制作表格、对比参数、分享方案 |
| 收集型玩家 | 查找地图点位、进度和遗漏内容 | 使用地图、清单和查询工具 |
| 体验服关注用户 | 获取更新、爆料、资格和开放时间 | 搜索资格、抢码、报名和版本信息 |
| 工具业务人员 | 判断什么值得做、先做什么 | 查看需求评分、证据和竞品情况 |

**核心场景示例：**

> 当三角洲行动玩家需要配置武器和满足战备值要求时，因为参数复杂且攻略分散，希望输入已有装备后得到明确配装建议，从而降低试错成本。

> 当体验服用户关注新版本测试时，因为更新、爆料和资格消息分散且时效短，希望集中查看当前节点、更新内容和报名时间，从而避免错过资格。

### 15.5 监控对象与范围

监控对象由游戏管理功能统一维护。当前示例包括三角洲行动、原神、鸣潮、火影忍者、洛克王国：世界、崩坏：星穹铁道、永劫无间手游、绝区零、失控进化和异环等。

仅“热门”“在运营”“测试中”的游戏进入挖掘流程；“已停运”游戏不再产生新需求，但保留历史数据。

**包含内容：**

- 配装、战备、机制计算、交互地图、抽卡分析、战绩查询等工具诉求；
- 攻略信息分散、参数复杂、反复提问等可被结构化解决的问题；
- 体验服更新、爆料、资格招募、开放时间和当前节点；
- 用户自制工具、表格、文档和外部平台同类工具；
- 站内已有工具的优化、更新和触达诉求。

**排除内容：**

- 仅提到“工具”但与当前游戏无关的内容；
- 普通游戏下载教程或无明确需求的宣传内容；
- 其他游戏的体验服信息误匹配到当前游戏；
- 仅有情绪或泛泛评价、无法还原用户目标的内容；
- 与工具和体验服业务无关的纯资讯。

### 15.6 数据来源和关键词

**数据来源：**

- TapTap；
- 抖音；
- 小黑盒；
- Tap+快爆论坛监控后台；
- 后续接入的其他公开平台。

**搜索配置示例：**

| 平台 | 关键词示例 | 抓取说明 |
|---|---|---|
| TapTap | 工具、体验服 | 全局应用于活跃游戏 |
| 小黑盒 | 工具 | 关注工具讨论和外部方案 |
| 抖音 | 洛克王国世界工具、三角洲行动工具、失控进化工具、体验服 | 关注重点游戏和热点内容 |

**需求词示例：**

- 有没有、求分享、怎么弄、哪里查、求推荐；
- 计算器、模拟器、地图、查询、助手、工具箱；
- 配装、战备、抽卡、概率、资源点、收集路线；
- 资格、招募、报名、申请、抢码、开放时间。

**排除和关联规则：**

- 内容必须能够关联当前游戏；
- 体验服对象原则上要求内容中出现当前游戏和体验服标识；
- 相同链接和同平台同标题内容进行去重；
- 已有站内工具名称用于匹配，不直接作为新需求。

### 15.7 需求识别规则

**工具需求有效信号：**

1. 多名玩家重复询问同一问题；
2. 攻略信息散落，用户需要跨平台整理；
3. 出现用户自制表格、文档、网页或小程序；
4. 游戏机制包含大量参数、组合或概率计算；
5. 外部平台已提供同类工具；
6. 内容热度高，且问题能够通过工具明确解决。

**体验服需求有效信号：**

1. 出现体验服、测试服、先遣服、共研服、内测或封测标识；
2. 内容涉及版本更新、玩法改动或测试公告；
3. 内容涉及新角色、新地图、新武器等爆料；
4. 内容涉及资格招募、报名入口、抢码和明确时间；
5. 可以判断当前为观察中、内容爆料、版本更新、预告中、报名中或已结束等节点。

### 15.8 分类、评分和优先级

**工具需求分类示例：**

- 配装/战备工具；
- 交互地图；
- 抽卡/概率分析；
- 资格/福利聚合；
- 机制计算器；
- 排行榜/对战数据；
- 剧情/收集进度；
- 攻略辅助；
- 模拟器；
- 数据库；
- 其他。

**体验服需求关注点：**

- 爆料内容；
- 更新内容；
- 资格招募。

**系统分析信号：**

- 重复提问密度；
- 信息分散度；
- 民间工具萌芽；
- 资格稀缺信号；
- 机制复杂度；
- 内容热度；
- 外部平台工具上线。

需求按潜力分划分为S、A、B、C四级。潜力分高不代表自动立项，还需结合工具化可行度、站内已有能力和业务优先级判断。

### 15.9 标准交付

**每日总览：**

- 今日需求数量；
- 活跃趋势游戏；
- 潜力分不低于70的高价值需求；
- 工具类型、需求等级和热门品类分布；
- 当日重点建议；
- 数据采集和分析完成状态。

**需求卡片：**

- 游戏及需求类别；
- 需求标题和描述；
- 潜力分、等级和工具化可行度；
- 七维需求信号；
- AI提炼的高频问题、信息缺口和判断理由；
- 证据帖及原始链接；
- 站内已上线工具匹配；
- 相似历史需求；
- 需求处理状态。

**体验服卡片：**

- 更新和爆料内容；
- 资格招募状态和时间；
- 当前体验服节点；
- 代表性证据内容。

**持续交付：**

- 历史需求排行榜；
- 按游戏、类型、等级、状态和日期筛选；
- 需求状态跟踪；
- 数据源新增内容提醒；
- 采集失败、数据不足和过滤原因展示。

### 15.10 协作与反馈

需求挖掘支持方负责数据采集、信号计算、需求提炼、证据整理、已有工具匹配和结果展示。

工具与体验服业务负责人负责：

- 确认重点游戏和优先权重；
- 审核需求是否有效；
- 决定采纳、开发、上线或驳回；
- 补充业务判断和备注；
- 反馈上线结果和业务效果。

需求状态统一为：

1. 待评估；
2. 已采纳；
3. 开发中；
4. 已上线；
5. 已驳回。

### 15.11 验收标准

1. 系统能够基于游戏列表和全局搜索词执行采集；
2. 同一游戏可产出多个不同主题的需求；
3. 不同游戏的内容不会错误混入当前游戏证据；
4. 工具需求与体验服需求能够正确分类；
5. 体验服需求能够展示更新、爆料、资格和当前节点；
6. 重点需求包含真实、去重且可访问的证据链接，无法访问时明确标注；
7. 已上线站内工具能够被识别并提示优先评估优化；
8. 需求能够被业务更新状态并进入历史沉淀；
9. 采集失败、重复、内容无关和平台返回不足等情况能够被解释；
10. 每日分析完成状态清晰，避免无意重复执行。

### 15.12 风险与后续优化

| 风险 | 处理方式 |
|---|---|
| 平台登录或风控导致采集失败 | 支持登录、代理、单项重试和失败原因展示 |
| 不同游戏名称相似导致串数据 | 使用游戏关联和体验服专属过滤规则 |
| AI分析结果过于宽泛 | 结合领域关键词、规则分析和证据校验 |
| 高热内容被误判为需求 | 要求同时满足具体问题或场景信号 |
| 站内已有工具被重复推荐 | 建立已上线工具库并执行匹配 |
| 业务状态未及时反馈 | 在需求管理中明确负责人和定期复盘 |
